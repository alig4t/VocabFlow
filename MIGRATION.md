# Migration Guide: MySQL to PostgreSQL

This document describes how to migrate the original MySQL database (containing a flat `words` table) to the new PostgreSQL schema used by the English Learning Platform.

---

## Original MySQL Schema

The legacy database contained a single `words` table with the following structure:

```sql
CREATE TABLE `words` (
  `id`              INT UNSIGNED     NOT NULL AUTO_INCREMENT,
  `eng`             VARCHAR(255)     NOT NULL,
  `per`             VARCHAR(255)     NOT NULL,
  `description`     TEXT             DEFAULT NULL,
  `example`         TEXT             DEFAULT NULL,
  `example_trs`     TEXT             DEFAULT NULL,
  `chapter`         TINYINT UNSIGNED DEFAULT NULL,
  `unit`            TINYINT UNSIGNED DEFAULT NULL,
  `is_known`        TINYINT(1)       NOT NULL DEFAULT 0,
  `created_at`      TIMESTAMP        NOT NULL DEFAULT CURRENT_TIMESTAMP,
  `updated_at`      TIMESTAMP        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

Key differences from the new schema:

| MySQL (legacy) | PostgreSQL (new) | Notes |
|---|---|---|
| `id` INT AUTO_INCREMENT | `id` TEXT (cuid) | Surrogate integer replaced by collision-resistant string ID |
| `example` TEXT | `primary_example` TEXT | Column renamed |
| `example_trs` TEXT | `primary_example_trs` TEXT | Column renamed |
| `chapter` TINYINT | `chapter` INT | Type widened |
| `unit` TINYINT | `unit` INT | Type widened |
| `is_known` TINYINT(1) | (removed) | Replaced by `user_word_progress` rows with `status` enum |
| No module concept | `module_id` FK | All migrated words belong to the vocabulary module |
| No multi-user support | `user_id` FK on progress | Progress is now per-user |

## Step 1: Export from MySQL

### Export the full database

```bash
mysqldump \
  --host=127.0.0.1 \
  --port=3306 \
  --user=<mysql_user> \
  --password \
  --single-transaction \
  --routines \
  --triggers \
  --default-character-set=utf8mb4 \
  english_learning > mysql_backup.sql
```

### Export only the words table as CSV (recommended for selective migration)

```bash
mysql \
  --host=127.0.0.1 \
  --user=<mysql_user> \
  --password \
  --database=english_learning \
  -e "SELECT id, eng, per, description, example, example_trs, chapter, unit, is_known, created_at FROM words ORDER BY id" \
  --batch \
  --silent \
  | sed 's/\t/,/g' > words_export.csv
```

Alternatively, export as SQL INSERT statements only:

```bash
mysqldump \
  --host=127.0.0.1 \
  --user=<mysql_user> \
  --password \
  --no-create-info \
  --skip-triggers \
  --complete-insert \
  english_learning words > words_only.sql
```

## Step 2: Data Transformation Notes

Before importing into PostgreSQL, account for the following differences:

### TINYINT to INT

MySQL `TINYINT(1)` and `TINYINT UNSIGNED` do not exist in PostgreSQL. The new schema uses plain `INTEGER`. No value transformation is required because the data fits comfortably in INT; only the column type declaration changes.

### Auto-increment INT to cuid strings

The legacy `id` is a numeric auto-increment. The new schema uses cuid strings generated by Prisma. Do not attempt to carry over the old integer IDs. Instead, let PostgreSQL and Prisma generate new IDs during the insert.

If you need a mapping between old and new IDs (e.g., for referential integrity in external systems), create a temporary translation table:

```sql
CREATE TABLE migration_id_map (
  old_mysql_id  INTEGER NOT NULL,
  new_pg_id     TEXT    NOT NULL,
  PRIMARY KEY (old_mysql_id)
);
```

### NULL handling

MySQL may store empty strings where `NULL` is more appropriate. Before migrating, normalize:

```sql
-- Run in MySQL before export
UPDATE words SET description   = NULL WHERE description   = '';
UPDATE words SET example       = NULL WHERE example       = '';
UPDATE words SET example_trs   = NULL WHERE example_trs   = '';
```

### Character encoding

Both the old and new schemas use UTF-8 / utf8mb4. Ensure the export uses `--default-character-set=utf8mb4` and the PostgreSQL database was created with `ENCODING 'UTF8'`.

### Timestamps

MySQL `TIMESTAMP` values are stored in UTC when `time_zone=+00:00`. PostgreSQL `TIMESTAMP WITH TIME ZONE` (mapped by Prisma as `DateTime`) is also UTC. Values transfer without conversion as long as the MySQL server timezone is UTC. If it is not, convert during export:

```sql
-- In MySQL: verify server timezone
SELECT @@global.time_zone, @@session.time_zone;
```

### `is_known` column

The boolean `is_known` flag is not directly migrated. After migration you can optionally recreate legacy progress records:

```sql
-- After words are migrated, create KNOWN progress entries for words
-- that had is_known = 1 in MySQL.
-- This requires the user's new PostgreSQL ID.
```

## Step 3: PostgreSQL Import Approach

### 3a. Create the database

```bash
createdb -U postgres english_learning
```

### 3b. Enable pgcrypto (needed for uuid generation in raw SQL; not required when using Prisma cuid)

```sql
CREATE EXTENSION IF NOT EXISTS "pgcrypto";
```

### 3c. Run Prisma migrations to create all tables

```bash
cd backend
npx prisma migrate deploy
```

This creates the full schema: `users`, `refresh_tokens`, `learning_modules`, `words`, `word_examples`, `user_word_progress`, `synonym_groups`.

### 3d. Insert the vocabulary learning module

```sql
INSERT INTO learning_modules (id, name, slug, description, is_active, "order", created_at)
VALUES (
  gen_random_uuid()::TEXT,
  '4000 Essential English Words',
  'vocabulary',
  'Core vocabulary for English learners',
  TRUE,
  1,
  NOW()
)
ON CONFLICT (slug) DO NOTHING;
```

Or use the Prisma seed script which does the same idempotently:

```bash
npx prisma db seed
```

## Step 4: Sample Migration SQL

The following statements show how to migrate rows from a staging table loaded with MySQL data into the final Prisma-managed tables.

### 4a. Load MySQL dump into a staging table

Create a temporary staging table that mirrors the old MySQL structure:

```sql
CREATE TEMPORARY TABLE mysql_words_staging (
  old_id       INTEGER,
  eng          TEXT,
  per          TEXT,
  description  TEXT,
  example      TEXT,
  example_trs  TEXT,
  chapter      INTEGER,
  unit         INTEGER,
  is_known     SMALLINT,
  created_at   TIMESTAMPTZ
);
```

Import the CSV produced in Step 1:

```bash
psql -U postgres -d english_learning \
  -c "\COPY mysql_words_staging (old_id, eng, per, description, example, example_trs, chapter, unit, is_known, created_at) FROM 'words_export.csv' CSV HEADER"
```

### 4b. Migrate words into the new schema

```sql
-- Capture the module ID for use in the INSERT
WITH vocab_module AS (
  SELECT id AS module_id FROM learning_modules WHERE slug = 'vocabulary'
)
INSERT INTO words (
  id,
  eng,
  per,
  description,
  primary_example,
  primary_example_trs,
  chapter,
  unit,
  module_id,
  created_at,
  updated_at
)
SELECT
  -- Generate a new cuid-compatible unique ID using pgcrypto
  encode(gen_random_bytes(16), 'hex'),
  s.eng,
  s.per,
  NULLIF(s.description,  ''),
  NULLIF(s.example,      ''),
  NULLIF(s.example_trs,  ''),
  s.chapter,
  s.unit,
  (SELECT module_id FROM vocab_module),
  COALESCE(s.created_at, NOW()),
  NOW()
FROM mysql_words_staging s
ON CONFLICT DO NOTHING;
```

### 4c. Populate migration ID map (optional but recommended)

```sql
INSERT INTO migration_id_map (old_mysql_id, new_pg_id)
SELECT
  s.old_id,
  w.id
FROM mysql_words_staging s
JOIN words w ON w.eng = s.eng
           AND w.chapter IS NOT DISTINCT FROM s.chapter
           AND w.unit    IS NOT DISTINCT FROM s.unit;
```

### 4d. Recreate legacy is_known progress (optional)

If you want to preserve which words were marked as known by the legacy single-user setup, and you have created a user in PostgreSQL first:

```sql
-- Replace '<user_id>' with the actual cuid of the target user in PostgreSQL
INSERT INTO user_word_progress (id, user_id, word_id, review_mode, status, updated_at)
SELECT
  encode(gen_random_bytes(16), 'hex'),
  '<user_id>',
  m.new_pg_id,
  'EN_TO_FA',
  'KNOWN',
  NOW()
FROM mysql_words_staging s
JOIN migration_id_map m ON m.old_mysql_id = s.old_id
WHERE s.is_known = 1
ON CONFLICT (user_id, word_id, review_mode) DO NOTHING;
```

## Step 5: Prisma Migration Commands Reference

| Command | Purpose |
|---|---|
| `npx prisma migrate dev` | Apply pending migrations in development (creates migration files) |
| `npx prisma migrate deploy` | Apply pending migrations in production (no schema changes, no prompts) |
| `npx prisma migrate reset` | Drop the database and reapply all migrations from scratch (development only) |
| `npx prisma migrate status` | Show which migrations have been applied vs. pending |
| `npx prisma db seed` | Run `prisma/seed.ts` to populate default data |
| `npx prisma generate` | Regenerate the Prisma Client after schema changes |
| `npx prisma studio` | Open the Prisma Studio GUI to inspect data |
| `npx prisma db pull` | Introspect an existing database and update `schema.prisma` |

## Step 6: Data Validation Queries

Run these queries after migration to confirm data integrity.

### Row count comparison

```sql
-- Count migrated words
SELECT COUNT(*) AS total_words FROM words;

-- Count words per chapter and unit
SELECT chapter, unit, COUNT(*) AS word_count
FROM words
GROUP BY chapter, unit
ORDER BY chapter, unit;
```

### Verify no orphaned records

```sql
-- Words without a valid module
SELECT COUNT(*) AS orphaned_words
FROM words w
LEFT JOIN learning_modules m ON m.id = w.module_id
WHERE m.id IS NULL;

-- Examples without a valid word
SELECT COUNT(*) AS orphaned_examples
FROM word_examples e
LEFT JOIN words w ON w.id = e.word_id
WHERE w.id IS NULL;

-- Progress records without a valid user or word
SELECT COUNT(*) AS orphaned_progress
FROM user_word_progress p
LEFT JOIN users u ON u.id = p.user_id
LEFT JOIN words w ON w.id = p.word_id
WHERE u.id IS NULL OR w.id IS NULL;
```

### Verify no NULL English or Persian values

```sql
SELECT COUNT(*) AS missing_eng FROM words WHERE eng IS NULL OR eng = '';
SELECT COUNT(*) AS missing_per FROM words WHERE per IS NULL OR per = '';
```

### Check for duplicate words within the same module, chapter, and unit

```sql
SELECT eng, chapter, unit, module_id, COUNT(*) AS duplicates
FROM words
GROUP BY eng, chapter, unit, module_id
HAVING COUNT(*) > 1;
```

### Verify enum values are valid

```sql
-- Should return 0 rows if all statuses are valid
SELECT *
FROM user_word_progress
WHERE status    NOT IN ('NOT_READ', 'KNOWN', 'NOT_KNOWN')
   OR review_mode NOT IN ('EN_TO_FA', 'FA_TO_EN');
```

### Sample spot-check

```sql
SELECT
  w.eng,
  w.per,
  w.chapter,
  w.unit,
  m.name AS module,
  COUNT(e.id) AS example_count
FROM words w
JOIN learning_modules m ON m.id = w.module_id
LEFT JOIN word_examples e ON e.word_id = w.id
GROUP BY w.id, w.eng, w.per, w.chapter, w.unit, m.name
ORDER BY w.chapter, w.unit, w.eng
LIMIT 20;
```
